# ARCHITECTURE.md — платформа отчётов (Report Platform)

Документ для защиты решения: компоненты, потоки данных, инструкция для команды, решения с альтернативами, ограничения прототипа. Код дополняет, но не заменяет этот текст.

---

## Соответствие тестовому заданию

| Требование | Как закрыто в прототипе |
|------------|-------------------------|
| API: запуск генерации | `POST /api/reports/:reportId/runs` — создаётся `ReportRun`, job в Redis |
| API: статус | `GET /api/runs/:id`, журнал — `GET /api/runs` |
| API: скачивание результата | `GET /api/runs/:id/download` (после `SUCCEEDED`) |
| API: список отчётов | `GET /api/reports` (каталог из `registry.ts`) |
| API: список запусков | `GET /api/runs` |
| Асинхронная генерация | Worker + BullMQ; API отвечает **202** с `runId`, не ждёт файла |
| UI | React (TS): выбор шаблона, запуск, журнал, скачивание, загрузка файла для отчёта `upload-csv` |
| Инфраструктура | `docker compose up --build` — Postgres, Redis, api, worker, frontend |
| ≥ 2 разных отчёта | Три шаблона с **разными источниками данных** (см. ниже); минимум задания перевыполнен |
| Разные форматы | XLSX и PDF в реестре; `upload-csv` и `weather-pdf` умеют PDF |
| Node + TS, React + TS | Да |
| Лицензии | Зависимости с открытыми лицензиями; шрифты Noto Sans — OFL |

**Источники данных (как в реальности будут разными):**

1. **Локальная БД** — отчёт `sales-xlsx`: агрегаты из Postgres (`Sale`).
2. **Внешний API** — отчёт `weather-pdf`: open-meteo.com, график/таблица в PDF.
3. **Файл пользователя** — отчёт `upload-csv`: загрузка табличного файла, нормализация имени, парсинг в worker; метаданные загрузки в таблице `FileUpload`, байты — на диске в volume.

Так видно паттерн «новый отчёт = новый модуль + запись в реестре», при этом тип источника не зашит в платформе.

## 1. Архитектура: компоненты, потоки, границы ответственности

### Компоненты

| Компонент | Ответственность |
|-----------|-----------------|
| **UI (`frontend`)** | Каталог шаблонов, запуск, polling статусов, ссылка на скачивание, загрузка файла для `upload-csv`, выбор формата XLSX/PDF где доступно. |
| **API (`api`)** | HTTP: отчёты, запуски, загрузки, выдача файлов из `FILES_DIR`; валидация; постановка job; **не** выполняет тяжёлую генерацию. |
| **Worker** | Чтение очереди, смена статуса `ReportRun`, генерация файла, запись результата в volume, обновление `outputPath` / ошибок. |
| **Postgres** | `ReportRun` (жизненный цикл запуска), `Sale` (демо), `FileUpload` (метаданные загрузки; содержимое файла не в БД). |
| **Redis** | Очередь BullMQ `report-jobs`. |
| **Volume `files`** | Исходные загрузки `uploads/{uploadId}/`, результаты `{runId}/...`. Общий для api и worker. |

Граница: **API быстрый и stateless по генерации**; долгая работа только в worker. Единая файловая среда — договорённость через `FILES_DIR` и один volume в compose.

### Поток данных (успешный запуск)

1. Пользователь в UI инициирует запуск → `POST /api/reports/:reportId/runs`.
2. API валидирует параметры (для `upload-csv` — строка `FileUpload` + файл на диске), создаёт `ReportRun` (`QUEUED`), кладёт job `{ runId }` в Redis, отвечает **202**.
3. Worker забирает job, переводит запуск в `RUNNING`, вызывает генератор отчёта.
4. Генератор читает свой источник (Prisma / HTTP / файл), пишет XLSX или PDF в `FILES_DIR/<runId>/`.
5. Worker сохраняет `SUCCEEDED`, `outputPath`, `outputName`.
6. UI по опросу видит статус и открывает `GET /api/runs/:id/download` — API **читает** `ReportRun` и отдаёт файл; **отдельной записи в БД при скачивании нет** (прототип; см. раздел про прод).

### Загрузка файла (только для `upload-csv`)

1. `POST /api/uploads` — файл на диск, строка `FileUpload`; при ошибке БД файл откатывается.
2. UI хранит `uploadId` и `fileName` до запуска отчёта.
3. Запуск с `params: { uploadId, fileName }` связывает запуск с уже сохранённой загрузкой.

## 2. Как добавить новый отчёт (пошагово для разработчика)

### Шаг 1 — Реестр

Файл `backend/src/reports/registry.ts`: новый `id`, `name`, `description`, `supportedFormats`, `defaultFormat`.

### Шаг 2 — Генератор

Файл `backend/src/worker/reports/<id>.ts`: асинхронная функция с контрактом как у соседей:

- вход: `run`, `filesDir`, `relDir` (префикс для относительного пути результата), `onProgress(pct, message)`;
- выход: `{ outputName, outputPath }` (пути в POSIX-стиле относительно `filesDir`).

Источник данных — на выбор: Prisma, fetch к API, файлы из `filesDir`, комбинация.

### Шаг 3 — Роутинг worker

`backend/src/worker/main.ts`: ветка `if (run.reportId === "<id>") { ... generate... }`.

### Шаг 4 — Параметры

Схема `zod` внутри генератора, разбор `run.params`. На API при необходимости — отдельная валидация в `POST .../runs` (как для `upload-csv`).

### Шаг 5 — UI

В `frontend/src/ui/App.tsx` при необходимости ветка `params` для нового `id` (сейчас параметры демо захардкожены).

### Шаг 6 — Проверка

`docker compose up --build`, прогон сценария в UI, контроль `QUEUED → RUNNING → SUCCEEDED` и скачивания.

### Паттерн «отчёт из загрузки» (`upload-csv`)

Используйте существующий поток:

1. `POST /api/uploads` — расширения и лимиты см. в `backend/src/api/main.ts`; новые форматы — парсер в `backend/src/lib/loadUploadTableRows.ts` и тесты рядом.
2. Метаданные пишутся в `FileUpload`; тело файла — в `FILES_DIR/uploads/{uploadId}/`. Для PDF с кириллицей нужны шрифты в `backend/fonts/` (Noto Sans, OFL), см. `pdfFont.ts` и существующие генераторы.
3. При другом отчёте «из файла» можно ссылаться на ту же таблицу или расширить `params` (например `runId → uploadId`).

## 3. Принятые решения и альтернативы (ключевые)

1. **BullMQ + Redis для очереди**  
   *Альтернативы:* RabbitMQ, NATS, SQS.  
   *Почему Redis:* одна зависимость с простым локальным запуском, достаточно для прототипа и защиты идеи «асинхронно».

2. **Отдельный процесс worker**  
   *Альтернативы:* генерация внутри запроса Fastify, отдельные serverless-функции.  
   *Почему отдельный сервис:* API не блокируется, масштабирование воркеров отдельно от HTTP — ближе к прод-форме «платформа отчётов».

3. **Postgres для состояния запусков и доменных демо-данных**  
   *Альтернативы:* только Redis, файловый журнал.  
   *Почему SQL:* история запусков, фильтры, связка с будущими пользователями/организациями без переписывания ядра.

4. **Результаты и загрузки на файловом volume, не BYTEA в БД**  
   *Альтернативы:* объектное хранилище сразу; хранение файла в Postgres.  
   *Почему volume:* один `docker compose`, общий каталог для api и worker; для задания достаточно. В проде — S3/MinIO и presigned URL.

5. **`FileUpload` в БД + файл на диске**  
   *Альтернативы:* только диск (риск «осиротевших» файлов без журнала); только БД (тяжело для больших таблиц).  
   *Почему гибрид:* аудируемая загрузка (размер, имя, время), проверка перед запуском отчёта, при этом тело таблицы не раздувает БД.

6. **Polling в UI вместо WebSocket**  
   *Альтернативы:* SSE/WebSocket.  
   *Почему polling:* меньше кода, понятно для демо; для прод — push или подписка на события.

7. **Fastify + Prisma**  
   *Альтернативы:* Nest, Express, Drizzle.  
   *Почему:* лёгкий сервер, явные маршруты, Prisma даёт миграции и типобезопасный доступ к схеме из api и worker.

## 4. Что не сделали и почему. Что добавили бы для продакшена

| Область | Прототип | Прод |
|---------|----------|------|
| Безопасность | Нет аутентификации, нет ACL по отчётам/файлам | OAuth2/OIDC, RBAC, аудит |
| Параметры отчётов | Частично зашиты в UI | Схемы параметров (OpenAPI/JSON Schema), динамические формы |
| Файлы | Один volume, нет TTL | S3/MinIO, lifecycle, антивирусное сканирование |
| Скачивание | Нет учёта факта скачивания | Таблица событий или счётчик `downloadCount` / signed URL с истечением |
| Загрузки в UI | Нет списка всех `FileUpload` | Админ/API `GET /uploads` с пагинацией |
| Надёжность | Базовые ретраи очереди | Идемпотентные ключи запуска, DLQ, алерты |
| Наблюдаемость | Логи в консоль | OpenTelemetry, метрики, трассировки |
| Тесты | Unit на утилиты (CSV, имена, ODS-цепочка) | Контрактные тесты API, e2e с тестовым compose |

Инструментарий ИИ использовался как ускоритель; архитектурные решения зафиксированы здесь, чтобы на защите было **что** и **зачем**, а не отсылка к «сгенерировалось само».
